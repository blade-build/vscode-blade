# 使用

## 目标浏览器

活动栏中的 **Blade** 视图以 **目录树** 形式展示工作区，从 `//` 根节点直到各个包。
展开 / 折叠状态会跨会话记忆；合成的外部库（`#pthread` 等）会被隐藏。

每个 **目标** 显示其规则类型与图标，并提供行内操作：

- **构建** —— 构建该目标。
- **运行** —— 针对 `*_binary` 目标（`blade run`）。
- **测试** —— 针对 `*_test` 目标（`blade test`）。
- **调试** —— 针对原生 `cc_binary` / `cc_test` 目标。
- **打开 BUILD 文件** —— 打开该包的 BUILD 文件。

每个 **目录**（以及 `//` 根）提供：

- **构建目录** / **测试目录** —— 通过 `//path/...` 模式递归构建或测试其下的全部内容。

## 活动目标与状态栏

状态栏显示活动目标以及 **构建 / 运行 / 测试 / 清理** 按钮。在浏览器中选择目标
（或执行 **Blade: Select Active Target**）会更新它。该选择按工作区记忆。

## 构建配置

目标视图标题栏中有一个 **release / debug** 选择器，它会把 `-p <profile>` 注入到
每次 构建 / 运行 / 测试 / 调试 中，默认 `release`，并按工作区记忆。用它可以在
优化构建与调试构建之间整体切换，而无需修改配置。

## 任务

所有执行都通过 VS Code 的 **Tasks API**，而非裸终端：

- 编译诊断由 `$gcc` 问题匹配器解析并显示在 **问题** 面板；
- 构建失败会返回非零退出码，因此损坏的目标不会被悄悄运行；
- blade 任务也会出现在 **终端 → 运行任务** 中（`blade: build …`、
  `blade: test …`、`blade: clean`）。

**运行** 使用 `blade run`，它会先构建再运行目标 —— 没有硬编码的产物路径，因此
不受构建配置或平台影响。

## 清理

**清理** 执行 `blade clean`（而非手动 `rm -rf`），从而遵循 Blade 自身的产物布局。
